# Receipts - CLAUDE.md

## What is Receipts
GitHub for non-coders. Voice note in, timestamped AI-structured claim out.
Captures the thinking, decisions and proof of work behind what non-technical
people build with AI. Individual proof of work + team collaboration +
org oversight.

Live at: https://get-receipts.lovable.app

## Supabase Schema
Table: entries
- id (uuid)
- owner_id (uuid, nullable)
- project_id (uuid, nullable)
- recorded_at (timestamptz)
- raw_transcript (text)
- claim (text)
- created_at (timestamptz)
- entry_type (text) - build/decision/idea/shipped/dropped/log
- tool_tags (text[])
- project_tag (text)
- user_id (uuid, nullable)
- transcript (text)
- user_name (text)

Table: waitlist
- id, email, created_at

Table: profiles
- id (uuid, FK to auth.users)
- display_name (text)
- ai_level (int)
- ai_level_title (text)
- role_category (text)
- company_context (text)
- personalised_tagline (text)
- onboarding_complete (bool)
- q1_answer (text)
- q2_answer (text)
- profile_public (bool)
- preferred_language (text, default 'en')
- workspace_id (uuid, FK to workspaces)

Table: ideas
- id, owner_id, raw_text, tag, project_id, created_at

Table: projects
- id, owner_id, name, one_liner, goal, status, created_at
- shipped_at (timestamptz) — set when generate-shipped-card runs
- key_wins (text[]) — 3 punchy wins, Claude-generated
- one_line_learning (text) — single learning sentence, Claude-generated
- tools_used (text[]) — tools/APIs extracted from entries
- copy_prompt (text) — Claude prompt for "copy this build"; written by
  generate-shipped-card at ship time and by copy_build MCP tool on demand
- RLS: owners can read/write their own; anon + authenticated can SELECT
  where status = 'Shipped'

Table: workspaces
- id, name, owner_id, created_at
- RLS: SELECT allowed if owner OR workspace_member; INSERT/UPDATE for owner

Table: workspace_members
- id, workspace_id, user_id, role (owner/member), joined_at
- UNIQUE(workspace_id, user_id)
- RLS: SELECT where user_id = auth.uid() (flat, no recursive subquery)
- INSERT where user_id = auth.uid()

Table: workspace_invites
- id, workspace_id, invite_code (unique, 8-char), created_by, created_at,
  expires_at (7 days)
- RLS: members can SELECT and INSERT

Table: workspace_feed
- id, workspace_id, user_id, project_id, event_type, created_at
- RLS: members can SELECT and INSERT

Edge functions deployed:
- process-entry (Claude Haiku, generates claim; --no-verify-jwt)
- assess-ai-level (Claude Haiku, returns ai_level 1-5 + profile fields; --no-verify-jwt)
- transcribe-audio (ElevenLabs scribe_v2 STT proxy; --no-verify-jwt)
- text-to-speech (ElevenLabs eleven_multilingual_v2 TTS proxy; --no-verify-jwt)
- generate-shipped-card (Claude Haiku; fetches project + entries, generates key_wins/
  one_line_learning/tools_used/copy_prompt, sets shipped_at + status='Shipped';
  posts to workspace_feed if owner has workspace_id; --no-verify-jwt)
- create-workspace (creates workspace + member + invite + updates profile; auth via
  Bearer token → supabase.auth.getUser(token); --no-verify-jwt)
- join-workspace (validates invite code, checks expiry, upserts member, updates
  profile workspace_id; --no-verify-jwt)
- receipts-mcp (MCP Streamable HTTP, 9 tools, Bearer JWT auth; --no-verify-jwt)

All edge functions: JWT verification disabled via supabase/config.toml + --no-verify-jwt flag
All edge functions: claude-haiku-4-5-20251001 (cost reduction from Sonnet)

Key technical note: edge functions use service role client +
supabase.auth.getUser(token) to identify caller — NOT auth.uid() which
only works in Postgres RLS context.

## MCP Integration — COMPLETE (April 30 2026)
Goal: Log entries directly from Claude conversations without opening the app

### What is built and working
- ✅ Local stdio MCP server at /mcp/src/index.ts — tested, entries log to Supabase
- ✅ Hosted edge function at supabase/functions/receipts-mcp/index.ts
  - MCP Streamable HTTP transport (JSON-RPC 2.0 over POST)
  - GET handler returns Protected Resource Metadata
  - 401 responses include WWW-Authenticate header
  - Mirrors client protocol version in initialize response
  - source: 'mcp' tag on all log_entry inserts
  - log_entry calls process-entry after insert to generate claim via Claude,
    then PATCHes the entry with claim + entry_type + project_tag
    (best-effort: raw entry is always saved even if claim generation fails)
- ✅ Deno Deploy proxy at mcp-proxy/main.ts — deployed at mcp.receipts.tools
  - Auto-deploys from GitHub: Receipts-demo/receipts-demo, entry mcp-proxy/main.ts
  - mcp-proxy/deno.json enables unstable KV API: {"unstable": ["kv"]}
  - Full OAuth 2.1 server built into the proxy (replaced Supabase OAuth beta)
  - Proxies all MCP traffic to the Supabase edge function
  - WWW-Authenticate rewritten to point to mcp.receipts.tools
- ✅ DNS: mcp.receipts.tools live with SSL cert via Deno Deploy + Lovable DNS
- ✅ OAuth 2.1 server endpoints (all in mcp-proxy/main.ts):
  - GET  /.well-known/oauth-protected-resource — points authorization_servers to proxy
  - GET  /.well-known/oauth-authorization-server — proxy's own discovery doc
  - POST /oauth/register — dynamic client registration, stores client in Deno KV
  - GET  /oauth/authorize — stores PKCE params in KV, redirects to receipts.tools/login
  - GET  /oauth/callback — receives Supabase JWT from login page, stores in KV,
         redirects to receipts.tools/oauth/consent?state=...
  - POST /oauth/approve — called by consent page with {state}, generates encrypted
         auth code (AES-GCM), returns {redirectTo} for window.location.href
  - POST /oauth/token — decrypts auth code to get payload, validates PKCE S256,
         returns Supabase JWT as access_token with expires_in: 3600
- ✅ Auth code is AES-GCM encrypted JSON — no KV lookup at token exchange time,
  completely avoids Deno Deploy cross-instance isolation issues
  - Payload: {access_token, code_challenge, code_challenge_method, client_id,
    redirect_uri, state}
  - Key derived from CODE_SECRET env var via SHA-256
- ✅ Deno KV (10 min TTL) used only for auth state during authorize→callback:
  key ["auth", state] stores PKCE params + access_token
- ✅ claude.ai connected end-to-end — MCP tools working in production

### copy_prompt format
Structured multi-line block written to projects.copy_prompt by both
generate-shipped-card (at ship time) and copy_build MCP tool (on demand):

  I want to build: {goal}

  Tools to use: {tools_used, comma-joined}   ← omitted if empty

  What worked:
  - {key_wins[0]}
  - {key_wins[1]}
  - {key_wins[2]}                            ← only items that exist

  Key learning: {one_line_learning}           ← omitted if empty

  Help me build my version.

### Deno Deploy env vars required
- SUPABASE_ANON_KEY — used to validate Supabase JWTs at /oauth/callback
- CODE_SECRET — any 32+ char random string, used to derive AES-GCM key

### OAuth flow (end-to-end)
1. claude.ai → GET /oauth/authorize (client_id, PKCE params)
2. Proxy stores params in KV, redirects to https://receipts.tools/login?oauth_redirect=...
3. Login page signs user in, redirects to https://mcp.receipts.tools/oauth/callback
   ?state=...&access_token={supabase_jwt}
4. Proxy validates token, stores in KV, redirects to
   https://receipts.tools/oauth/consent?state=...
5. Consent page shows "Claude wants to connect" — user clicks Allow
6. Consent page POSTs {state} to https://mcp.receipts.tools/oauth/approve
7. Proxy generates encrypted auth code, returns {redirectTo} pointing to claude.ai
8. Consent page does window.location.href = redirectTo
9. claude.ai POSTs to /oauth/token with code + code_verifier
10. Proxy decrypts code, validates PKCE, returns Supabase JWT as access_token
11. claude.ai uses JWT as Bearer token on all MCP calls
12. receipts-mcp resolveUser() validates JWT via Supabase /auth/v1/user

### New user OAuth flow (added May 2 2026)
New users arriving from claude.ai MCP connector can create an account inline:
1. Login page detects oauth_redirect param → shows "Connect to Claude" heading
   + Sign in / Sign up tab strip
2. Sign up tab: creates account via supabase.auth.signUp, then redirects to
   /onboarding?oauth_redirect={encoded}
3. Onboarding completes → finishAndRedirect helper checks oauth_redirect →
   gets session → window.location.href = ${oauth_redirect}&access_token=${token}
4. All three onboarding exit points (Log build, See profile, Skip) go through
   finishAndRedirect so the OAuth flow always completes

### Lovable frontend contract
Login page (/login):
- Detects oauth_redirect param; shows Sign in / Sign up tabs when present
- Sign in path: after auth, checks onboarding_complete; sends incomplete users
  to /onboarding?oauth_redirect=... first
- Sign up path: after account creation, sends to /onboarding?oauth_redirect=...
- Already-logged-in: useEffect fires immediately, same onboarding check applies

Onboarding page (/onboarding):
- Reads oauth_redirect from URL search params
- finishAndRedirect(fallback) helper used at all 3 exit points:
  if oauth_redirect → supabase.auth.getSession() →
    window.location.href = `${oauth_redirect}&access_token=${token}`
  else → navigate(fallback)

Consent page (/oauth/consent) — POST state to proxy, redirect on approval:
  const state = new URLSearchParams(window.location.search).get('state');
  // Allow button:
  const res = await fetch('https://mcp.receipts.tools/oauth/approve', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ state }),
  });
  const { redirectTo } = await res.json();
  window.location.href = redirectTo;
  Note: read 'state' from URL, NOT 'authorization_id' (that was the old Supabase flow)

## Key Product Decisions Made
- ElevenLabs chosen for both STT (scribe_v2) and TTS (eleven_multilingual_v2)
  — multilingual, auto-detect, better than Web Speech API on mobile
- Audio sent as base64 JSON to edge functions to avoid Deno FormData parsing issues
- speak() uses raw fetch (not supabase.functions.invoke) to get binary audio
- Haiku model on all edge functions — sufficient quality, lower cost/latency
- Language-agnostic: removed German toggle, ElevenLabs + Claude auto-handle all languages
- localhost for multi-user without auth — replaced by Supabase Auth (April 15 2026)
- Lovable for UI, Claude Code for backend logic
- Five prompt types chosen based on real user need: built/learnt/solved/problem/next
- Tool auto-detection by Claude preferred over manual tagging
- Renamed "My Projects" to "My Builds"
- Entry types map to grouping tabs
- Record page shows inbox model (unassigned only), not all entries
- Workspace feed uses parallel Supabase lookups (not join) to avoid FK alias issues
- Edge functions identify caller via supabase.auth.getUser(Bearer token) with
  service role client — SUPABASE_ANON_KEY is NOT auto-injected into edge functions
- copy_build MCP tool has no owner_id filter — teammates can copy any project
  (get_entries keeps owner_id filter; raw entries stay private)
- search_builds filters in Deno code, not PostgREST, because tools_used is text[]
  and PostgREST cs filter only does exact element match, not substring
